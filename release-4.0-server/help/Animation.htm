OPCAT animation for 1st time users

 

 

 

 

 

 

 

 

 

 

 

            This manual is meant to be an introduction guide for 1st time users of OPCAT animation plug-in.  We’ll go through the modeling and animation of a simple system step by step to give the user a short review of OPCAT animations possibilities.

 

            The guide assumes previous knowledge of OPM syntax and principals. A more extensive and deep review of animations rules is available too and recommended for users planning to use OPCAT for more complex project modeling.

           

 

                        The first step will be to simply model a system to run on animations plug-in. lets take a house committee payments system for example, it is simple enough and yet implements many animation features.

 

House committee payments system model:

 

                        For this model we’ll give system just one task: deciding which letter to send each neighbor in the building. There will be three options:

 

1)     Full payment required– in case the neighbor forgot to pay.

2)     Partial payment required – in case the bill was partially paid or neighbor deserves some discount.  

3)     Receipt – in case the bill was already paid.

The system should add entry to bills report-log so it can be verified afterwards.

 

 

Let’s see the list of objects we’ll need to model the system:

 

 

1)     Chairman of committee – to run the mailing system.

2)     Database – to keep billing data.

3)     Report entry – a new entry to bills logging.

4)     Mails – the e-mails actually sent to neighbor.

 

Let’s try to identify the roles of these objects:

 

            The chairman is in charge of the flow and initiates it; therefore he is the agent handling the process. The Database is used by the system to decide which type of mail to send to each neighbor, therefore it is an instrument. The mails are abstract objects of information created by the system according to the flow of the process for specific neighbor, so is the new entry to bills log.

 

            Accordingly to the above conclusions we should have the following objects in SD:

1)     Chairman: environmental; physical.

2)     Database: environmental; informatical.

3)     Report entry: systematical; informatical.

4)     Mails: systematical; informatical.

1)     The system should be given ID of current neighbor.

Now that we have a model ready to test let’s take a look at animations interface:

Animations interface has three parts:

 

1)     Menu and operating buttons: here you operate the animation and change animations preferences through main menu.

2)     Animation window: here you see the animation running. You can highlight different ‘things’ in window in order to activate or deactivate them, but you can’t change the model in animation interface.
At the bottom of animations window you can see the status bar which shows the running status and step.

3)     OPD browsing window: here you can browse through different SDs to check their state or change activation of ‘things’.

 

What will happen during the simulation?

 

Basically OPCAT will just run through the flow of our model activating and deactivating ‘things’ according to creation, consumption and other links.  OPCAT will follow the logic combination of conditions and flow order and flow accordingly.

 

How do I see what’s happening?

 

            You’ll see the inactive ‘things’ colored white and getting darker as they’re being created , in case of objects, or being executed, in case of processes. The flow between the links will be noted with red tokens running through the links the flow reached.

Now we’ll take a closer look at operating buttons and menu and learn how to run our models animations.

 

     

 

      Play: this button starts/resumes the animation (but doesn’t activate agent links!)

 

     

 

      Back: takes the animation back a specific number of steps.

 

     

 

      Number of steps to go back when ‘back’ button pressed.

 

     

 

      Forward: takes the animation forward a specific number of steps.

 

     

 

Number of steps to go forward when ‘forward’ button pressed.

 

 

Pause: pauses/ resumes continuous to “step-by-step” animation.

 

 

Stop: stops animation.

 

 

 

      Activation: activates highlighted ‘thing’.

 

     

 

      Deactivation: deactivates highlighted ‘thing’.

 

     

 

      Zoom in and out (not through OPD tree but visually).

 

     

 

      Exit animation mode.

 

     

 

      Preferences: opens the menu.

 

Now let’s check the menu:

 

 

Here you can see different adjustable preferences for the animation settings.

 

We’ll check them one by one:

-          Step duration: duration of every animation step in milliseconds.

-          Process duration: duration of process in number of steps per process.
Duration can be a fixed number of steps or uniformly distributed between a lower and an upper bound.

-          Reaction time: number of steps for object to react to action. Same options as in process duration.

-          Default object instances: weather at default only a single instance of an object should exist or many should be created.

-          Use automatic initiation: if not checked you’ll have to initiate everything needed manually through all steps.

-          Automatic move between OPD: if checked it will zoom-in and out between OPD with the flow of model. Else it will run normally but you’ll see only the OPDs you open manually.

-          Animation mode: animation can run continuously alone or step-by-step activating next step only when “forward” button is pressed.

-          Random state selection: if checked, while creating an object with multiple states it will choose initial state randomly.

Now that we had run an example and know how it basically works let’s make a little summary of how to use OPCAT animation:

 

1)     Create model.

2)     Setup preferences.

3)     Press ‘play’.

4)     Activate necessary agents.

5)     Run it!
